--- BUGS / FINDINGS DISCOVERED WHILE BUILDING AND RUNNING THIS ---

1. pqcrypto's verify() argument order bug in the EXISTING R2 code.
   pqcrypto.sign.falcon_512.verify and pqcrypto.sign.ml_dsa_44.verify are both
   verify(public_key, message, signature). crypto_test.py and pqc_benchmark.py from R2
   both call verify(public_key, signature, message) -- arguments swapped. This does not
   raise an exception, it silently returns False every time. This means every "verify"
   timing number quoted in the R1/R2 material was timing an always-failing call, not a
   genuine verification. Worth fixing in those scripts and re-measuring verify time
   properly before quoting it again.

2. Gating the SD Options/OfferService path is necessary but not sufficient in someipy
   2.1.2. The library's _handle_send_event_request forwards an event to every current
   subscriber of a (service_id, instance_id, eventgroup_id) the moment ANY locally
   connected UDS client calls send_event() for it -- it never checks whether that
   client's claimed src_endpoint matches whichever OfferService was actually accepted.
   In other words, in this specific someipy version, a rogue ECU can inject a malicious
   frame straight into existing subscribers over the daemon's local socket API without
   ever getting its own OfferService accepted into the routing table at all. This is
   why the first flood-attack run above still showed 65% malware even with the
   OfferService gate already active -- and it's why secure_daemon.py gates
   _handle_send_event_request too, not just datagram_received_mcast. This is worth a
   sentence in the R3 report: the real, exploitable delivery path in this
   implementation is event-forwarding, and the SD-spoofing story in R1/R2 is the right
   mental model for the protocol but someipy's own architecture adds this second path.

3. Environment quirk (only matters if testing on a loopback-only machine, e.g. a
   minimal container): sending to a multicast address without first setting
   IP_MULTICAST_IF explicitly fails with "OSError: Network is unreachable" if there is
   no default route configured. Fixed in camera_service.py and attacker_service.py by
   setting IP_MULTICAST_IF to 127.0.0.1 before sending. Should be harmless on a normal
   machine with a real network interface, but if you hit this error, this is why.
